# Datasheet: BBO Capstone Project Dataset

## Motivation

### Why was the dataset created?

This dataset was created as part of the **Imperial College Professional Certificate in Machine Learning & Artificial Intelligence** Black-Box Optimization (BBO) Capstone Challenge. The task requires maximizing 8 unknown functions over a series of weekly submission rounds, using only observed input-output pairs. The dataset supports surrogate model training and acquisition function optimization within a Bayesian Optimization framework.

### Who created it?

The dataset was created by **Benjamin Baumann** as part of the capstone project. The course framework (initial random samples, function evaluation infrastructure, and submission system) was provided by Imperial College London.

### Was it funded or supported by an organisation?

The project is part of an academic programme at Imperial College London. No external funding was involved.

## Composition

### What does the dataset contain?

The dataset consists of **8 separate CSV files**, one per black-box function, containing observed input-output pairs collected over 11 weeks of optimization. Each file records:

- **Input features** (`x0`, `x1`, ..., `xN`): continuous values in the unit hypercube [0, 1]^d
- **Output** (`y`): a scalar real-valued function evaluation
- **Source label** (`source`): provenance tracking (e.g., `initial`, `week_4_submission`)

### Data format

Each CSV file uses the format:

```
x0,x1,...,xN,y,source
0.630734,0.621812,1.7727099645980484,week_8_submission
```

- Inputs: 6 decimal places
- Outputs: full floating-point precision
- Source: string label indicating provenance

### Are there any gaps?

- **No missing values**: Every observation has complete input, output, and source information.
- **Spatial coverage gaps**: The data is heavily concentrated around regions near current best values due to the exploitation-heavy strategy adopted in later weeks. Large portions of the input space (especially in higher-dimensional functions) remain unsampled.
- **No PII or sensitive content**: All data points are evaluations of synthetic mathematical functions.

### Subpopulations

The observations fall into two categories:
1. **Initial samples** (`source: initial`): randomly generated starting points provided by the course framework
2. **Weekly submissions** (`source: week_N_submission`): strategically generated query points from the optimization pipeline

## Collection Process

### How were the data collected?

Data collection occurred in two phases:

1. **Initial phase**: Random samples were generated by the course framework using an undisclosed sampling strategy. These form the baseline dataset for each function (typically 10-40 points depending on dimensionality).

2. **Optimization phase** (Weeks 1-11): One query per function per week was generated using an evolving Bayesian Optimization pipeline:
   - **Weeks 1-3**: Gaussian Processes with Upper Confidence Bound (UCB) acquisition
   - **Week 4**: Neural network ensemble surrogates with gradient-guided optimization
   - **Weeks 5-6**: Breakthrough discovery via grid search and trust region exploration
   - **Week 7**: NeurIPS 2020 BBO techniques (TuRBO, multi-kernel GP, Thompson Sampling)
   - **Weeks 8-10**: Multi-kernel GP ensemble (Matern kernels) with Probability of Improvement (PI) selection, trust regions, and hard constraints
   - **Week 11**: HEBO-inspired upgrades (output warping via Yeo-Johnson, noisy EI, multi-acquisition ensemble)

### Sampling strategy

Queries were generated deterministically (seed=42) using a surrogate-model-guided approach. Candidate points were sampled within trust regions (local neighbourhoods around promising areas) using Sobol quasi-random sequences, then ranked by acquisition function scores. The strategy evolved from broad exploration (early weeks) to tight exploitation (later weeks).

### Time frame

- **Duration**: 11 weeks (January-February 2026)
- **Frequency**: 1 query per function per week
- **Feedback delay**: Results returned after each weekly submission round

### Ethical review

No ethical review was required. The dataset involves evaluations of synthetic mathematical functions with no human subjects, personal data, or real-world impact.

## Preprocessing and Uses

### Preprocessing applied

The raw data in `samples.csv` files is **unprocessed** — it contains the original input coordinates and function evaluations as returned by the evaluation system.

During model training within the notebooks, the following reversible transformations are applied:
- **Input standardization**: `sklearn.preprocessing.StandardScaler` centres and scales input features to zero mean and unit variance
- **Output standardization**: Outputs are similarly scaled for numerical stability in GP fitting
- These transformations are applied at runtime and do not modify the stored data

### Intended uses

- Training surrogate models (Gaussian Processes, neural networks) for Bayesian Optimization
- Evaluating acquisition function strategies for sample-efficient optimization
- Studying exploration-exploitation trade-offs under extreme budget constraints
- Academic analysis of black-box optimization techniques

### Inappropriate uses

- **Generalizing to other optimization problems** without adapting the strategy to the new problem's characteristics
- **Treating the data as representative** of the full function landscape — the observations are heavily biased toward promising regions
- **Inferring function structure** from this sparse data alone — the dataset captures only a tiny fraction of the input space

## Distribution and Maintenance

### Where is the dataset available?

The dataset is stored in the project's GitHub repository under `data/function_{1-8}/samples.csv`. The full repository includes notebooks, source code, and documentation alongside the data.

### Terms of use

This dataset is part of an academic coursework project. It is shared for educational and portfolio purposes. The underlying black-box functions are proprietary to the Imperial College BBO Challenge and are not included.

### Maintenance

The dataset is maintained by the project author. It is updated weekly as new function evaluations are received (one new row per function per week). The `source` column tracks the provenance of each observation, enabling clear version history.

After Week 13 (the final round), the dataset will be frozen and no further updates will be made.
